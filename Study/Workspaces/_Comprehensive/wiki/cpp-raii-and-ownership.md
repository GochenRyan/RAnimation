# C++ RAII、所有权与资源生命周期

## 1. 这篇专题为什么存在

`d533911 fix RAII` 这次提交的价值，不在于“又讲了一遍 RAII 定义”，而在于它暴露了一个更贴近工程现实的问题：

- 资源对象能不能拷贝
- 容器里怎么安全构造资源
- 代码当前是在拥有资源，还是只是在借用资源
- 初始化接口到底有没有把所有权表达清楚

所以这篇专题不把 RAII 当成教科书术语，而是把它放回当前仓库的语境里，和所有权、借用、移动语义、失败路径一致性一起看。

## 2. 这次提交到底修了什么

这次提交主要修了三类问题。

### 2.1 资源对象不该被拷贝，却存在值拷贝风险

看 [RenderData.h](/S:/Github/RAnimation/Samples/SkeletalAnimation/include/Model/RenderData.h#L52)，`RTextureData` 里持有的是典型的资源句柄：

- `nri::Texture*`
- `nri::Memory*`
- `nri::Descriptor*`
- `nri::DescriptorSet*`

这些成员不是“普通值类型字段”，而是资源所有权边界的一部分。

这次提交给它补上了：

- 删除拷贝构造
- 删除拷贝赋值
- 保留默认移动构造
- 保留默认移动赋值

这不是语法修饰，而是在告诉调用方：

- 这个类型不是普通可复制数据
- 它应该被移动，而不是被无意识复制

### 2.2 容器中的资源对象不应该靠临时对象绕一圈再插入

看 [Mesh.cpp](/S:/Github/RAnimation/Samples/SkeletalAnimation/src/Model/Mesh.cpp#L86) 和 [Model.cpp](/S:/Github/RAnimation/Samples/SkeletalAnimation/src/Model/Model.cpp#L64)。

原来的写法大致是：

- 先构造一个局部 `RTextureData newTex`
- 把资源加载到这个临时对象
- 再 `insert` 到 map 里

这个模式的问题是：

- 生命周期表达不直观
- 一旦类型不允许拷贝，就会开始和容器插入语义冲突
- 失败路径下，临时对象和容器状态的关系也更绕

新写法改成：

- `try_emplace` 先在容器里占位
- 拿到容器中对象的引用
- 直接向这个最终归属位置写入资源
- 失败时再把该条目擦掉

这让“资源最终归谁拥有”变得清楚很多。

### 2.3 借用已有资源时，不应该复制整个资源对象

看 [Model.cpp](/S:/Github/RAnimation/Samples/SkeletalAnimation/src/Model/Model.cpp#L273) 和 [Model.cpp](/S:/Github/RAnimation/Samples/SkeletalAnimation/src/Model/Model.cpp#L326)。

原来的模式是：

- 从 `mTextures` 里找一张纹理
- 找到后，把 `RTextureData` 整体拷贝到一个局部变量 `diffuseTex`

这对资源对象来说很危险，因为语义上这里根本不是“我要拥有一份新纹理”，而只是：

- “我想临时借用这张已经存在的纹理”

新写法改成：

- `RTextureData* diffuseTex = nullptr;`
- 找到时保存指针

这更准确表达了语义：

- 当前代码不拥有这张纹理
- 只是借用已有对象

### 2.4 初始化接口要把所有权变化表达清楚

看 [SkinningPipeline.h](/S:/Github/RAnimation/Samples/SkeletalAnimation/include/Renderer/SkinningPipeline.h#L15)。

这次把：

```cpp
nri::Pipeline* pipeline
```

改成了：

```cpp
nri::Pipeline*& pipeline
```

这里的重点不是“引用比指针高级”，而是接口语义发生了澄清：

- 调用方传进来的不是一个普通只读指针值
- 被调函数会修改这个指针，让它指向一个新创建出来的资源

也就是说，这个接口在表达：

- “我要把新资源交回给调用方”

这比单纯传值指针更接近真实语义。

## 3. RAII 在这个仓库里到底该怎么理解

如果只把 RAII 理解成：

- “构造拿资源，析构释放资源”

会太浅。

对这个仓库来说，更有用的理解是：

- 资源必须绑定到一个明确的拥有者对象上
- 一旦对象不是普通值类型，就必须显式设计拷贝、移动和借用语义

也就是说，RAII 在这里真正解决的不是“有没有析构函数”，而是下面四个问题：

- 谁拥有资源
- 谁只是借用资源
- 资源是否允许被复制
- 失败路径下资源状态是否仍然一致

这也是为什么 `RTextureData` 这次虽然没有自定义析构逻辑变化，但仍然属于 RAII 修复的一部分：

- 因为它的所有权语义被收紧了

## 4. 这次提交抽象出的 5 条 C++ 工程规则

### 4.1 持有底层资源句柄的对象，默认先审查是否应该禁止拷贝

只要一个类型里保存的是这类成员：

- 裸资源句柄
- GPU 资源指针
- descriptor / memory / pipeline 句柄
- 和底层生命周期强绑定的对象

就不要默认把它当普通可复制值类型。

更合理的默认起点通常是：

- 删除拷贝
- 允许移动

然后再根据实际需要放开。

### 4.2 容器里的资源对象，优先原地构造或先占位再填充

如果资源最终就归容器所有，就不要绕一圈：

- 先局部构造
- 再插入容器

更清晰的方式是：

- `emplace`
- `try_emplace`
- 先占位，再对容器里那个最终对象写入内容

这样更容易保持：

- 生命周期一致
- 失败路径一致
- 所有权表达一致

### 4.3 查表后只想读取已有资源时，优先用指针或引用，不要值拷贝

如果你的语义是：

- “我想借用这个对象”

那表达方式就应该像借用：

- 指针
- 引用
- `const` 引用

而不是把整个资源对象复制一份。

否则代码会在语义上撒谎：

- 表面像“得到了一份自己的资源”
- 实际上只是“想临时访问别人拥有的资源”

### 4.4 out-parameter 如果要把新资源交回调用方，接口必须把这一点写清楚

如果函数会创建资源并交回给调用方，接口就要明确表达：

- 这个参数会被写回
- 调用方之后要接管它

在当前仓库里，`nri::Pipeline*&` 就属于这种表达方式。

不是所有 out-parameter 都必须写成引用到指针，但必须让语义清楚：

- 这是输入
- 还是输出
- 还是输入输出

### 4.5 使用 move-only、指针、引用、`try_emplace`，不等于自动正确

这些技术手段只是表达语义的工具，不是正确性的保证。

真正要检查的是：

- 资源最终归谁
- 失败时容器是否保持一致
- 借用是否越界
- 调用顺序是否仍然有效

也就是说：

- 语法正确不等于所有权设计正确

## 5. 三个典型模式

这次提交里，最值得沉淀的是三个可复用模式。

### 5.1 模式一：资源对象默认 move-only

典型形式：

```cpp
T(const T&) = delete;
T& operator=(const T&) = delete;
T(T&&) noexcept = default;
T& operator=(T&&) noexcept = default;
```

适用场景：

- 类型里持有句柄
- 类型的复制语义不自然
- 复制后容易变成双重释放、悬空引用、状态不一致

### 5.2 模式二：容器拥有资源时，在容器里完成最终构造

典型思路：

- `try_emplace`
- 取得 `it->second`
- 直接初始化这个最终对象
- 若失败，清理对应条目

这种模式比“局部临时对象 + insert”更适合资源型对象。

### 5.3 模式三：借用资源时，显式使用指针 / 引用

典型思路：

- 容器里对象长期存在
- 当前函数只读或临时访问
- 用 `T*` 或 `T&` 表达借用

这比值拷贝更接近真实语义，也更适合 move-only 类型。

## 6. 这次提交其实不只是在修 RAII

如果只把这次改动命名为“RAII 修复”，会低估它真正涉及的范围。

更准确地说，它同时在修三件事：

- RAII
- 所有权设计
- 资源生命周期表达

其中：

- `delete copy / default move` 是对象语义层
- `try_emplace` 是容器归属层
- 指针借用替代值拷贝是访问语义层
- `Pipeline*&` 是接口所有权表达层

所以这篇专题的重点不是“析构函数怎么写”，而是：

- 如何让资源型代码的语义和真实生命周期一致

## 7. 本仓库中的实用检查表

以后你 review 这类代码，可以直接按下面的问题检查。

### 7.1 类型层

- 这个类型里是否持有裸句柄或资源指针？
- 如果持有，它真的应该可拷贝吗？
- 是否应该是 move-only？

### 7.2 容器层

- 这个资源最终归容器所有吗？
- 如果归容器所有，是否在容器内完成最终构造？
- 失败路径下容器状态是否仍然一致？

### 7.3 访问层

- 当前代码是在拥有资源，还是借用资源？
- 如果只是借用，是否错误地做了值拷贝？
- 指针 / 引用的生存期是否覆盖当前使用区间？

### 7.4 接口层

- 这个参数是输入、输出，还是输入输出？
- 接口签名有没有把资源交接关系表达清楚？
- 调用方能不能从签名直接看出所有权变化？

## 8. 最终结论

这次 `fix RAII` 最有价值的地方，不是“用了 RAII”这四个字，而是它把代码从“看起来能工作”推进到了“生命周期语义更清楚”。

如果把这次提交压缩成一句工程结论，就是：

- 对资源型 C++ 代码，真正重要的不是会不会写析构函数，而是能不能把拥有、借用、移动、失败路径这四件事表达清楚。

这才是当前仓库里最值得复用的 C++ 经验。
