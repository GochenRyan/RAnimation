# 静态库与动态库

## 1. 最小定义

先只记两句话：

- 静态库：库里的实现代码在链接期并入最终产物。
- 动态库：最终程序在运行期依赖一个独立的动态库文件。

这里最容易犯的错是按“来源”分类：

- “它是第三方库，所以应该做成动态库”

这是错的。真正决定静态还是动态的，不是它是不是第三方库，而是它在你的工程里扮演什么角色。

## 2. 三阶段模型

理解静态库和动态库，最稳的方式不是背术语，而是按三个阶段看：

### 2.1 编译期

编译期的核心工作是：

- 预处理
- 语法检查
- 把 `.cpp` 编成 `.obj`

最小链路：

```text
.cpp -> .obj
```

这一阶段主要关心：

- 头文件能不能找到
- 宏和编译选项是否一致
- 类型声明是否正确

通常还没有发生“把库真正装进最终程序”这件事。

### 2.2 链接期

链接期才是静态库和动态库差异真正显性的地方。

链接器会把多个 `.obj` 和库组合成最终产物，比如：

```text
main.obj + some_lib -> app.exe
```

#### 静态库在链接期做什么

静态库本质上是很多 `.obj` 的打包。

链接器会：

- 从静态库里按需取出需要的目标代码
- 把这些代码并入最终的 `exe` 或 `dll`

所以静态库的意义不是“运行时模块”，而是“链接期可复用的实现代码仓库”。

#### 动态库在链接期做什么

动态库真正的实现通常留在 `.dll` 里。

但链接器在构建 `exe` 时，通常不会把 `.dll` 内容直接拷进去。它依赖的是一个导入库，也常常叫 `.lib`。

这个导入库主要告诉链接器：

- 哪些符号来自某个动态库
- 最终程序在运行时要去哪个 `dll` 找这些实现

所以动态库的链接期结果不是“把实现拷进去”，而是“建立运行期依赖关系”。

### 2.3 运行期

#### 静态库

如果最终 `exe` 已经成功链接，运行时通常不再需要静态库文件本身。

因为链接器已经把需要的代码并入最终产物。

#### 动态库

动态库不一样。运行时必须能找到对应的 `dll`。

如果找不到，常见结果是：

- 程序启动失败
- 首次调用到该模块时报错
- 系统提示缺少某个 `dll`

所以动态库比静态库多出来的，不是“一个名字不同的文件”，而是一整套运行期部署和装载要求。

## 3. Windows 下的产物到底是什么

Windows 下最容易让人困惑的是：静态库和动态库都可能看到 `.lib`。

### 3.1 静态库的常见产物

- `xxx.lib`
- `xxx.pdb`（可选，调试信息）

这里的 `xxx.lib` 通常就是静态库本体，里面装的是实际目标代码。

### 3.2 动态库的常见产物

- `xxx.dll`
- `xxx.lib`
- `xxx.pdb`

这里要分清：

- `xxx.dll`：运行期真正需要加载的模块
- `xxx.lib`：链接期使用的导入库（import library）
- `xxx.pdb`：调试信息

所以一句必须记住的话是：

- 同样叫 `.lib`，有时它是静态库本体，有时它只是某个 `dll` 的导入库。

这就是为什么只看文件后缀，不足以判断“它到底是静态库还是动态库”。

## 4. 为什么静态库已经编进 exe 了，还需要 `.lib`

这是最关键的一个问题。

答案是：

- 静态库 `.lib` 的主要职责在链接期，不在运行期。

可以把过程拆开看：

```text
foo.cpp -> foo.obj
bar.cpp -> bar.obj
foo.obj + bar.obj -> MyLib.lib
main.obj + MyLib.lib -> app.exe
```

这里的 `MyLib.lib` 是什么？

- 它是很多 `.obj` 的打包
- 它给链接器提供“可复用的一组实现代码”

为什么需要它？

- 因为这样别的目标不用自己单独管理一堆 `.obj`
- 库可以被多个最终目标复用
- 构建系统和链接器都更容易组织依赖

所以更准确的说法是：

- 对静态库来说，`.lib` 是链接期中间产物
- 对运行程序来说，通常不再需要它
- 对继续开发、重新构建、重新链接来说，它仍然是重要产物

也就是说：

- “编进 exe 以后，它对运行期通常没用”是对的
- “所以它完全没用，可以永久删掉”是不对的

因为你下次重新链接还需要它。

## 5. 动态库相比静态库，多维护了什么

动态库不是“只是多一个 `.dll` 文件”。

它相比静态库，通常多维护了下面几类东西。

### 5.1 导出边界

静态库通常不需要特别声明“哪些符号对外导出”。

动态库通常要额外处理：

- 哪些类 / 函数 / 变量是公开 API
- 哪些实现细节不应该导出
- Windows 下的导出宏，比如：
  - `__declspec(dllexport)`
  - `__declspec(dllimport)`

### 5.2 运行期部署规则

静态库通常只要链接成功即可。

动态库还要明确：

- `dll` 拷贝到哪里
- 运行目录怎么组织
- Debug / Release 是否分开
- 程序启动时怎么找到这些 `dll`

### 5.3 ABI 和内存边界

只要开始跨 `dll` 边界，很多问题都会更敏感：

- 是否跨库传 STL 容器
- 谁负责分配和释放内存
- 是否跨库抛异常
- 编译器 / CRT / 编译选项是否一致

这些在静态库场景里通常没那么尖锐，因为最终代码进了同一个产物。

### 5.4 版本兼容

静态库升级通常意味着重新编译和重新链接。

动态库如果想单独替换，就要更在意：

- 新旧接口是否兼容
- 导出符号是否稳定
- 主程序是否仍能加载它

### 5.5 构建系统规则

动态库通常还要多写：

- `SHARED`
- 导出宏定义
- post-build copy
- install / package 规则
- 有时还包括运行路径相关配置

所以动态库的额外成本，本质上不是“多写几行代码”，而是开始显式维护模块边界。

## 6. 什么时候优先静态，什么时候考虑动态

### 6.1 优先静态库的典型场景

- sample
- 小工具
- 内部 helper
- 不需要运行期独立部署
- 不需要单独升级或替换该模块
- 更看重构建和部署简单性

这类场景里，静态库通常更直接。

### 6.2 考虑动态库的典型场景

- 插件系统
- 多个程序共享同一个运行时模块
- 需要运行期按需加载
- 需要把模块作为独立分发单元
- 希望不重链主程序就能替换某个模块

这类场景里，动态库的“独立性”才有明确收益。

所以真正该问的不是：

- “这个库能不能做成静态库？”

很多时候技术上都能。

真正该问的是：

- “我需不需要它在运行期保持独立模块身份？”

如果不需要，静态库通常更省事。

## 7. 回到本仓库的 `ImGuiFileDialog`

这次案例正好能说明一个关键点：

- “它是第三方库”并不等于“应该做成动态库”。

对当前仓库来说，`ImGuiFileDialog` 更像：

- sample 进程内的 UI helper
- 和 ImGui 紧耦合的一部分界面功能
- 不是一个需要独立部署、独立升级、独立替换的运行时模块

再结合源码现状：

- 它头文件里预留了 `IGFD_API` 宏
- 但默认这个宏是空的
- 它没有内建完整的 Windows DLL 导出逻辑
- 头文件注释里还明确说过，共享库用法不推荐，ABI 不保证

所以对当前仓库最合理的结论不是：

- “它不能做动态库”

而是：

- “当前工程没有足够收益去承担它作为动态库的额外复杂度”

静态库能满足需求，动态库只会引入更多运行期边界和部署问题。

## 8. 最终检查表

以后你再遇到“某个库到底该静态还是动态”，先按这 5 个问题检查：

1. 这个库是内部依赖，还是独立分发模块？
2. 运行期是否需要单独携带和加载 `dll`？
3. 当前问题发生在编译期、链接期，还是运行期？
4. 我看到的 `.lib` 是静态库本体，还是 `dll` 的导入库？
5. 这个库类型是显式声明的，还是被 `BUILD_SHARED_LIBS` 之类的全局设置间接影响的？

如果这 5 个问题能答清楚，大多数“静态库 / 动态库”的混乱都会消失。
