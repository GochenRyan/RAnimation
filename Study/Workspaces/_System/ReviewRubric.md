# Review Rubric

本文件定义学习材料批改标准。所有 `reviews/*.md` 应尽量按本标准给出可执行反馈。

## 批改目标

- 判断是否真正掌握，而不是只看是否“写了很多”
- 优先检查理解深度、证据链和边界意识
- 对错误结论要明确指出，不做模糊鼓励式评价

## 五个评分维度

每项建议使用 `0-5` 分：

### 1. 正确性

- 是否给出正确结论
- 是否存在关键事实错误
- 是否把局部现象误判成通用规律

### 2. 证据性

- 是否引用了源码、日志、调用链、结构体、函数关系
- 证据是否真的支撑结论
- 是否只写抽象描述而没有落到仓库事实上

### 3. 调用链与数据流

- 是否能把入口、关键函数、关键数据结构串起来
- 是否明确“谁创建、谁更新、谁消费、谁销毁”
- 是否能解释时序与依赖关系

### 4. 边界条件与风险

- 是否说明适用范围
- 是否意识到例外情况、默认行为、隐含前提
- 是否指出潜在失败点

### 5. 表达清晰度

- 结构是否清楚
- 段落是否围绕问题
- 是否便于复查和继续追问

## 批改结论等级

### `Pass`

- 关键结论正确
- 证据链基本成立
- 可以继续到下一主题

### `Revise`

- 有理解基础
- 但在关键证据、调用链或边界条件上不够扎实
- 需要改写后再进入下一主题

### `Rework`

- 关键结论错误
- 或回答缺乏仓库证据
- 或没有真正回答题目核心

## 单题批改建议格式

```md
# <题目 ID>

## Verdict

- Pass / Revise / Rework

## Scores

- Correctness: x/5
- Evidence: x/5
- Call Path and Data Flow: x/5
- Boundaries and Risks: x/5
- Clarity: x/5

## Findings

- 先写最重要的问题
- 再写次要问题

## Corrections

- 给出正确理解
- 指出应补充的源码和证据

## Next Step

- 明确要求补什么
```

## 批改时的硬性要求

- 发现核心错误时必须直接指出
- 必须区分“结论错了”和“证据不够”
- 若用户提供了 `artifacts/` 中的手推、日志、截图，必须纳入批改
- 若回答超出题目范围但核心没答到，按核心缺失处理
